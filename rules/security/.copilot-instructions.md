# Security Rules — Path-Specific Custom Instructions

## Purpose

This directory contains **15 SonarQube security vulnerability rules**. Every file defines a single rule that detects an exploitable weakness in application code. The assistant should treat these rules as high-stakes: inaccurate guidance here can directly impact application security posture.

---

## Category Conventions

| Property | Expected Value |
|---|---|
| `type` | `VULNERABILITY` (always) |
| `severity` | `CRITICAL` or `BLOCKER` for injection / RCE / data-exposure rules; `MAJOR` for lower-impact issues (e.g., insecure random, CSRF) |
| `impacts[].softwareQuality` | `SECURITY` (always) |
| `impacts[].severity` | `HIGH` for most rules; `MEDIUM` for lower-impact rules |
| `tags` | Must include `"security"` and `"owasp-top-10"` when applicable |

## Guidance for Creating New Security Rules

1. **Always use `"type": "VULNERABILITY"`.** Security rules are never `CODE_SMELL` or `BUG`.
2. **Default to `CRITICAL` severity.** Downgrade to `MAJOR` only if the vulnerability requires unlikely preconditions or has limited blast radius. Use `BLOCKER` for rules that can lead to full system compromise (RCE, authentication bypass, unrestricted data access).
3. **Tag with relevant standards.** Always include `"security"`. Add `"owasp-top-10"` if the vulnerability maps to an OWASP Top 10 category. Add attack-specific tags like `"injection"`, `"rce"`, `"phishing"`, `"authentication"`.
4. **Remediation examples must show the fix, not just describe it.** Use real API calls (e.g., `PreparedStatement`, `ProcessBuilder`, `SecureRandom`), not pseudocode.
5. **Explain the attack vector in the description.** State what an attacker can do if the vulnerability is present, not just what the code looks like.
6. **Remediation cost calibration for this category:**
   - 10–15min: Drop-in API replacement (e.g., `Random` → `SecureRandom`, adding cookie flags)
   - 20–30min: Requires input validation/sanitization logic or parameterized queries
   - 45min+: Architectural changes (e.g., switching serialization frameworks, SSRF allowlists)
7. **Debt function** should be `CONSTANT_ISSUE` — security fixes are per-occurrence, not proportional to code size.
8. **No `params` are typical** for security rules (vulnerabilities are binary: present or not). Only add params if there is a meaningful configurable threshold.

## Common Pitfalls

- **Don't conflate severity levels.** `severity` and `defaultSeverity` must always match. `impacts[].severity` is independent — it describes the quality impact, not the rule's trigger threshold.
- **Don't forget the attack-surface context.** A rule detecting `Math.random()` used for a coin-flip game is not a vulnerability. Descriptions should specify the security-sensitive context (token generation, cryptographic operations, authentication).
- **Don't use generic remediation.** "Use a safe API" is not helpful. Show the specific safe API with a compilable code snippet.
