# Security Rules — Copilot Custom Instructions

You are assisting with **security-focused SonarQube rule definitions** in this directory. Every rule here detects exploitable vulnerabilities or security weaknesses in source code. Follow these instructions precisely when creating, editing, or explaining security rules.

---

## Category Constraints

- **`type` MUST be `"VULNERABILITY"`** — all rules in this directory represent security vulnerabilities, never `CODE_SMELL` or `BUG`.
- **`severity` MUST be `"CRITICAL"` or `"BLOCKER"`** — security vulnerabilities are high-priority by nature. Use `BLOCKER` for issues with direct, trivially exploitable attack vectors (e.g., hardcoded credentials, unparameterized SQL). Use `CRITICAL` for issues requiring some preconditions to exploit (e.g., insecure cookies, weak cryptography). Only use `MAJOR` in rare cases where the security impact is indirect or contextual (e.g., insecure random for non-cryptographic use).
- **`impacts[].softwareQuality` MUST be `"SECURITY"`** with severity `"HIGH"` for BLOCKER/CRITICAL rules, or `"MEDIUM"` for MAJOR rules.
- **`tags` MUST include `"security"`** as the first tag. Include `"owasp-top-10"` for rules that map to an OWASP Top 10 category. Add domain-specific tags (e.g., `"sql"`, `"injection"`, `"cryptography"`, `"cookies"`, `"deserialization"`).

| Key | Detects |
|---|---|

## Security-Specific Guidance

### Writing Descriptions

Security rule descriptions must clearly articulate:
1. **The vulnerability** — what the insecure pattern is.
2. **The attack vector** — how an attacker can exploit it.
3. **The impact** — what damage results (data breach, RCE, privilege escalation, etc.).

Example: _"Detects potential SQL injection vulnerabilities where user input is directly concatenated into SQL queries without proper sanitization or parameterization."_

### Writing Remediation Examples

Security rule remediation `before`/`after` examples must:
- Show a **realistic, exploitable** non-compliant pattern in `before` (not just a comment describing the problem).
- Show a **concrete, secure** fix in `after` using industry-standard mitigations (parameterized queries, encoding functions, security libraries, etc.).
- Use **Java conventions** by default (matching SonarQube's ecosystem), but note the language if targeting another.
- Never include actual secrets, real API keys, or real credentials — use obvious placeholder values like `"mySecretPassword123"`.

### Remediation Cost

Security fixes typically have moderate constant cost. Use `CONSTANT_ISSUE` debt function. Typical ranges:
- Simple API swap (e.g., `Random` → `SecureRandom`): `"10min"`
- Code restructuring (e.g., parameterized queries, input validation): `"15min"` – `"30min"`
- Architecture change (e.g., implementing CSRF framework): `"30min"` – `"60min"`

### OWASP Top 10 Mapping

When a rule maps to an OWASP Top 10 category, include the `"owasp-top-10"` tag. Common mappings:
- **A01 Broken Access Control** — path-traversal, open-redirect, csrf-vulnerability
- **A02 Cryptographic Failures** — weak-cryptography, hardcoded-credentials, insecure-random
- **A03 Injection** — sql-injection, command-injection, ldap-injection, xss-vulnerability, xml-external-entity
- **A04 Insecure Design** — insecure-cookie, sensitive-data-exposure
- **A05 Security Misconfiguration** — xml-external-entity
- **A08 Software and Data Integrity Failures** — insecure-deserialization
- **A10 Server-Side Request Forgery** — server-side-request-forgery

### Parameters

Security rules generally do **not** have configurable `params` — an SQL injection is always an SQL injection regardless of thresholds. Only add `params` if there is a genuinely configurable aspect (e.g., a regex pattern for credential detection).

## Template for New Security Rules

```json
{
  "key": "rule-key-here",
  "name": "Human-Readable Vulnerability Name",
  "description": "Detects [vulnerability pattern] where [unsafe input/configuration] allows [attack vector], potentially leading to [impact].",
  "severity": "CRITICAL",
  "type": "VULNERABILITY",
  "tags": ["security", "relevant-domain-tag", "owasp-top-10"],
  "remediation": {
    "constantCost": "30min",
    "examples": [
      {
        "before": "// Realistic non-compliant code showing the vulnerability",
        "after": "// Secure compliant code using proper mitigation"
      }
    ]
  },
  "impacts": [
    {
      "softwareQuality": "SECURITY",
      "severity": "HIGH"
    }
  ],
  "defaultSeverity": "CRITICAL",
  "status": "READY",
  "debt": {
    "function": "CONSTANT_ISSUE",
    "offset": "30min"
  }
}
```

## Validation Checklist

Before saving a new or modified security rule, verify:
- [ ] `type` is `"VULNERABILITY"`
- [ ] `severity` is `"CRITICAL"` or `"BLOCKER"` (not lower without strong justification)
- [ ] `tags` array starts with `"security"` and includes `"owasp-top-10"` if applicable
- [ ] `impacts[0].softwareQuality` is `"SECURITY"`
- [ ] `description` explains the vulnerability, attack vector, and impact
- [ ] `remediation.examples` show realistic vulnerable code and a concrete fix
- [ ] `key` matches the filename (without `.json`), using lowercase hyphen-separated format
- [ ] No overlap with the 15 existing rules listed above
- [ ] JSON is valid and all required fields are present
- [ ] No real secrets, credentials, or sensitive data in examples
