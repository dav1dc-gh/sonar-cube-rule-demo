# Maintainability Rules — Copilot Custom Instructions

You are assisting with **maintainability-focused SonarQube rule definitions** in this directory. Every rule here addresses long-term code health — readability, modularity, documentation, naming consistency, and architectural clarity that determine how easily software can be understood, modified, and extended over time. Follow these instructions precisely when creating, editing, or explaining maintainability rules.

---

## Category Constraints

- **`type` MUST be `"CODE_SMELL"`** — all maintainability rules represent design or readability issues, never `VULNERABILITY` or `BUG`.
- **`severity` ranges from `"MINOR"` to `"MAJOR"`** — use `MAJOR` for structural issues that significantly impede comprehension or modification (e.g., circular dependencies, shotgun surgery, deep nesting). Use `MINOR` for cosmetic or convention issues that primarily affect consistency (e.g., inconsistent naming, missing javadoc, excessive comments). Never use `BLOCKER` or `CRITICAL` for maintainability rules.
- **`impacts[].softwareQuality` MUST be `"MAINTAINABILITY"`** — this is the defining quality dimension for this category. Severity should be `"HIGH"` for architectural issues, `"MEDIUM"` for structural readability issues, `"LOW"` for naming and documentation issues.
- **`tags` MUST include `"maintainability"`** as the first tag. Add relevant concern tags: `"readability"`, `"naming"`, `"conventions"`, `"architecture"`, `"coupling"`, `"documentation"`, `"design"`, `"refactoring"`, `"complexity"`.

## Maintainability-Specific Guidance

### Distinguishing from Code Smells

Maintainability rules and code smell rules both use `"type": "CODE_SMELL"`, but they focus on different concerns:

- **Code smells** (`rules/code-smells/`) focus on **object-oriented design anti-patterns** — structural problems at the class and method level (god class, feature envy, duplicate code, dead code).
- **Maintainability** (`rules/maintainability/`) focuses on **long-term code health across the codebase** — readability, documentation, naming, configuration management, dependency structure, and architectural modularity.

If a new rule is primarily about OO design violations or refactoring patterns (Extract Class, Replace Conditional), it belongs in `code-smells/`. If it's about readability for future developers, configuration hygiene, documentation, naming conventions, or dependency architecture, it belongs here.

### Subcategories

Maintainability rules fall into distinct subcategories. When creating a rule, identify which sub-concern it addresses:

- **Readability & Comprehension** — deep-nesting, long-methods, boolean-blindness, excessive-comments. Focus: reducing cognitive load for developers reading the code.
- **Naming & Conventions** — inconsistent-naming, missing-javadoc. Focus: consistency and self-documenting code.
- **Configuration & Externalization** — hardcoded-urls. Focus: separating configuration from code for deployment flexibility.
- **Dependency & Architecture** — circular-dependencies, hidden-dependencies, shotgun-surgery. Focus: clean module boundaries and low coupling.
- **Defensive Design** — missing-null-check, too-many-parameters, boolean-blindness. Focus: API design that prevents misuse.

### Writing Descriptions

Maintainability descriptions must articulate:
1. **What the rule detects** — the specific code pattern or practice.
2. **The long-term cost** — how this pattern makes the code harder to understand, modify, or extend over time.
3. **Who is affected** — typically future developers (including future-self), code reviewers, or new team members onboarding.

Example: _"Detects code with excessive nesting levels (if statements, loops, try blocks), which reduces readability and increases cognitive complexity."_

Avoid vague statements like "this is bad practice." Be specific about the measurable consequence: harder to test, increases change risk, obscures control flow, forces readers to hold too many states in working memory.

### Writing Remediation Examples

Maintainability remediation focuses on **improving readability and reducing cognitive load**:
- `before` examples should show code that is **technically correct but hard to read, understand, or modify** — the smell is not in the behavior but in the structure.
- `after` examples should show the same logic expressed **more clearly** — using guard clauses, meaningful names, extracted helper methods, configuration files, or explicit parameter objects.
- For architecture-level rules (circular dependencies, shotgun surgery), it is acceptable to use descriptive comments or pseudocode since the full fix may span multiple files.
- Use Java conventions by default.

### Configurable Parameters (`params`)

Maintainability rules benefit from threshold parameters when the boundary between "acceptable" and "problematic" is subjective:
- **Nesting depth**: default 4 (teams doing functional programming may tolerate less)
- **Method length**: varies widely by team convention
- **Parameter count**: common defaults are 5–7

Use `"type": "INTEGER"` for thresholds. Always provide a default that represents a widely accepted industry standard.

### Debt Functions

Most maintainability rules use `CONSTANT_ISSUE` since the fix effort per occurrence is predictable:
- Rename/convention fix: `"5min"`
- Add documentation: `"10min"` – `"15min"`
- Reduce nesting, extract method: `"15min"` – `"20min"`
- Externalize configuration: `"15min"` – `"30min"`
- Resolve circular dependency, refactor API: `"30min"` – `"60min"`

Use `LINEAR` only when the fix effort truly scales with a measurable dimension (e.g., lines in a method, number of parameters to refactor into an object).

## Template for New Maintainability Rules

```json
{
  "key": "rule-key-here",
  "name": "Human-Readable Maintainability Issue Name",
  "description": "Detects [pattern] that [reduces readability / increases coupling / hinders modification], making the code [harder to understand/modify/extend] for [future developers / new team members].",
  "severity": "MAJOR",
  "type": "CODE_SMELL",
  "tags": ["maintainability", "relevant-concern-tag", "readability"],
  "remediation": {
    "constantCost": "15min",
    "examples": [
      {
        "before": "// Code that is correct but hard to read, modify, or understand",
        "after": "// Same logic expressed more clearly using better structure"
      }
    ]
  },
  "impacts": [
    {
      "softwareQuality": "MAINTAINABILITY",
      "severity": "MEDIUM"
    }
  ],
  "defaultSeverity": "MAJOR",
  "status": "READY",
  "debt": {
    "function": "CONSTANT_ISSUE",
    "offset": "15min"
  }
}
```

For rules with configurable thresholds, add `params`:

```json
{
  "params": [
    {
      "key": "maxDepth",
      "name": "Maximum Allowed Depth",
      "description": "Maximum nesting depth before the rule triggers",
      "defaultValue": "4",
      "type": "INTEGER"
    }
  ]
}
```

## Validation Checklist

Before saving a new or modified maintainability rule, verify:
- [ ] `type` is `"CODE_SMELL"` (never `VULNERABILITY` or `BUG`)
- [ ] `severity` is `"MINOR"` or `"MAJOR"` (never `BLOCKER` or `CRITICAL`)
- [ ] `tags` array starts with `"maintainability"`
- [ ] `impacts[0].softwareQuality` is `"MAINTAINABILITY"`
- [ ] `description` explains the pattern, its long-term cost, and who is affected
- [ ] `remediation.examples` show technically correct but hard-to-maintain code, paired with a clearer alternative
- [ ] The rule belongs here, not in `code-smells/` (see distinguishing guide above)
- [ ] If the rule has a threshold, `params` is defined with industry-standard defaults
- [ ] `key` matches the filename (without `.json`), lowercase hyphen-separated
- [ ] No overlap with the 12 existing rules listed above
- [ ] JSON is valid and all required fields are present
