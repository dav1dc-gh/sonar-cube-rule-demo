# Maintainability Rules — Path-Specific Custom Instructions

## Purpose

This directory contains **12 SonarQube maintainability rules**. Each rule detects an issue that degrades long-term code health — making code harder to read, modify, test, or onboard new developers to. These rules complement the `code-smells` category but focus specifically on **readability, structure, and developer experience** rather than object-oriented design patterns.

---

## Category Conventions

| Property | Expected Value |
|---|---|
| `type` | `CODE_SMELL` for most rules; `BUG` for missing-null-check (because it causes runtime failures) |
| `severity` | `MAJOR` for structural issues (circular deps, hidden deps, shotgun surgery, long methods, null checks, too many params); `MINOR` for style/readability issues |
| `impacts[].softwareQuality` | `MAINTAINABILITY` (primary); `RELIABILITY` for missing-null-check |
| `tags` | Must include `"maintainability"`. Add concern-specific tags: `"readability"`, `"design"`, `"documentation"`, `"naming"`, `"configuration"` |

## Guidance for Creating New Maintainability Rules

1. **Distinguish from code-smells.** The `code-smells` directory covers OOP design patterns (god class, feature envy, refused bequest). This directory covers **structural readability and developer experience**: naming, nesting, documentation, configuration, parameter design, null safety. If the rule is about a classic OO "smell" from Fowler's refactoring catalog, it likely belongs in `code-smells/`.
2. **`BUG` type is rare but valid.** Use `BUG` only when the maintainability issue directly causes runtime failures (e.g., `missing-null-check` → `NullPointerException`). All other rules should be `CODE_SMELL`.
3. **Severity calibration:**
   - `MAJOR` — Structural issues requiring significant refactoring: circular dependencies, hidden dependencies, shotgun surgery, missing null checks, deep nesting, long methods, too many parameters.
   - `MINOR` — Style and convention issues that matter at scale but don't individually impede comprehension: naming, comments, documentation, hardcoded URLs, boolean blindness.
4. **Use `LINEAR` debt when fix effort scales with problem size.** Long methods use `LINEAR` because a 200-line method takes more work to refactor than a 101-line method. Shotgun surgery uses `LINEAR` because more scattered files = more effort. Most other rules use `CONSTANT_ISSUE`.
5. **Add `params` for numeric thresholds.** Any rule with a "maximum allowed X" concept should expose it as a parameter: max depth, max lines, max parameters, max boolean args.
6. **Remediation examples should be practical.** Show realistic code transformations, not abstract descriptions. For architectural rules (circular dependencies, hidden dependencies), show before/after class structures.
7. **Remediation cost calibration for this category:**
   - 5–10min: Rename, add null check, extract URL to config
   - 15–25min: Add documentation, refactor boolean params to enum/builder, break up parameters
   - 30–60min: Introduce dependency injection, break circular dependencies
   - 60min+: Decompose shotgun surgery with events/annotations, major structural refactor

## Relationship to Code Smells

These two categories share the `CODE_SMELL` type and the `MAINTAINABILITY` quality impact but address different concerns:

| Aspect | Code Smells (`rules/code-smells/`) | Maintainability (`rules/maintainability/`) |
|---|---|---|
| **Focus** | OOP design patterns, structural anti-patterns | Readability, developer experience, code health |
| **Origin** | Fowler's refactoring catalog, design pattern violations | Coding standards, team conventions, API ergonomics |
| **Typical fix** | Extract class/method, change inheritance to composition | Rename, add docs, externalize config, add null guard |
| **Overlap risk** | `long-parameter-list` (code-smells, max 5) | `too-many-parameters` (maintainability, max 7) |

When creating a new rule, check both directories to avoid duplication. If a rule exists in `code-smells/` with a different framing, consider whether the new rule adds distinct value or should just be a threshold adjustment on the existing one.

## Common Pitfalls

- **Don't duplicate `code-smells/long-parameter-list`.** The `too-many-parameters` rule here has a higher threshold (7 vs 5) and different framing (readability vs design smell). When creating parameter-count rules, be clear about which threshold and perspective applies.
- **Don't use `VULNERABILITY` type.** Even `hardcoded-urls` is a maintainability concern here, not a security vulnerability. If a hardcoded URL contains credentials, that belongs in `rules/security/hardcoded-credentials.json`.
- **Don't forget `defaultSeverity`.** It must always match `severity`. This is easy to overlook when copying from an existing rule with a different severity level.
- **Keep `impacts[].severity` independent from `severity`.** Rule severity (MAJOR/MINOR) is the trigger priority. Impact severity (HIGH/MEDIUM/LOW) describes the effect on software quality. A `MINOR` rule can still have `MEDIUM` maintainability impact if it compounds across a codebase.
