# Code Smells Rules — Copilot Custom Instructions

You are assisting with **code smell SonarQube rule definitions** in this directory. Every rule here identifies design problems, structural weaknesses, and code quality issues that make software harder to understand, test, and evolve. Follow these instructions precisely when creating, editing, or explaining code smell rules.

---

## Category Constraints

- **`type` MUST be `"CODE_SMELL"`** — all rules in this directory represent design and quality issues, never `VULNERABILITY` or `BUG`.
- **`severity` ranges from `"MINOR"` to `"MAJOR"`** — code smells are not crashes or security holes. Use `MAJOR` for design problems that significantly hinder development (e.g., god classes, high complexity). Use `MINOR` for stylistic issues or minor anti-patterns (e.g., magic numbers). Never use `BLOCKER` or `CRITICAL` for code smells.
- **`impacts[].softwareQuality`** should be `"MAINTAINABILITY"` (most common) or `"RELIABILITY"` for smells that increase bug likelihood (e.g., empty catch blocks).
- **`tags` MUST include `"code-smell"`** as the first tag. Add design principle tags where applicable (e.g., `"solid"`, `"design"`, `"refactoring"`, `"complexity"`).

## Code-Smell-Specific Guidance

### Design Principle Alignment

Code smell rules are rooted in established software design principles. When writing descriptions, reference the underlying principle being violated:

- **Single Responsibility Principle (SRP)** — god-class, feature-envy
- **DRY (Don't Repeat Yourself)** — duplicate-code, data-clumps
- **KISS (Keep It Simple)** — complex-methods, speculative-generality
- **Law of Demeter** — message-chains, feature-envy
- **Clean Code / Refactoring patterns** — magic-numbers, dead-code, empty-catch-block, primitive-obsession, refused-bequest
- **Interface Segregation / Encapsulation** — long-parameter-list, data-clumps

### Writing Descriptions

Code smell descriptions must explain:
1. **What the smell is** — the structural or design pattern being detected.
2. **Why it matters** — concrete consequences (harder to test, understand, modify, or extend).
3. **The design principle violated** — reference SRP, DRY, Law of Demeter, etc., where applicable.

Example: _"Detects classes that have grown too large and handle too many responsibilities, violating the Single Responsibility Principle and making the code difficult to understand and maintain."_

### Writing Remediation Examples

Code smell remediation follows **refactoring patterns**. Examples should:
- Show the **smell pattern** in `before` — a recognizable anti-pattern, not just a comment.
- Show the **refactored solution** in `after` — applying a named refactoring technique where possible (Extract Method, Extract Class, Replace Primitive with Object, etc.).
- Be realistic enough to illustrate the problem, but concise enough to fit in a JSON string field.
- For large-scale refactorings (god class, feature envy), it is acceptable to use descriptive comments rather than full code when the refactoring is too large to show inline.

### Configurable Parameters (`params`)

Code smell rules **frequently benefit from configurable thresholds**. Add `params` when:
- A rule detects something that exceeds a numerical limit (complexity, line count, parameter count, nesting depth).
- Different teams may have different tolerance levels for the smell.

Parameter conventions:
- Use `"type": "INTEGER"` for threshold parameters.
- Provide a **sensible default** that matches common industry standards (e.g., cyclomatic complexity = 15, max class lines = 500).
- Name parameters descriptively: `"Maximum Lines"`, `"Complexity Threshold"`, `"Maximum Methods"`.

### Debt Functions

Code smells use both debt function types:
- **`CONSTANT_ISSUE`** — for smells with a fixed remediation cost per occurrence (e.g., removing dead code: `"offset": "10min"`).
- **`LINEAR`** — for smells where remediation effort scales with the issue size (e.g., reducing complexity: `"coefficient": "1min"` per unit of complexity over the threshold). When using `LINEAR`, always set `"offset": "0min"`.

Typical remediation costs:
- Quick fix (rename, remove dead code, add logging to catch block): `"5min"` – `"15min"`
- Moderate refactoring (extract method, replace magic number): `"15min"` – `"30min"`
- Significant restructuring (split god class, eliminate duplication): `"1h"` – `"4h"`

## Template for New Code Smell Rules

```json
{
  "key": "rule-key-here",
  "name": "Human-Readable Smell Name",
  "description": "Detects [structural pattern] that [violates principle / causes problem], making the code [harder to test/maintain/understand].",
  "severity": "MAJOR",
  "type": "CODE_SMELL",
  "tags": ["code-smell", "relevant-design-tag"],
  "remediation": {
    "constantCost": "30min",
    "examples": [
      {
        "before": "// Non-compliant code showing the smell",
        "after": "// Refactored code applying the appropriate pattern"
      }
    ]
  },
  "impacts": [
    {
      "softwareQuality": "MAINTAINABILITY",
      "severity": "MEDIUM"
    }
  ],
  "defaultSeverity": "MAJOR",
  "status": "READY",
  "debt": {
    "function": "CONSTANT_ISSUE",
    "offset": "30min"
  }
}
```

When the rule has a configurable threshold, add the `params` field and use the `LINEAR` debt function:

```json
{
  "debt": {
    "function": "LINEAR",
    "coefficient": "1min",
    "offset": "0min"
  },
  "params": [
    {
      "key": "threshold",
      "name": "Threshold Name",
      "description": "Description of what the threshold controls",
      "defaultValue": "10",
      "type": "INTEGER"
    }
  ]
}
```

## Validation Checklist

Before saving a new or modified code smell rule, verify:
- [ ] `type` is `"CODE_SMELL"` (never `VULNERABILITY` or `BUG`)
- [ ] `severity` is `"MINOR"` or `"MAJOR"` (never `BLOCKER` or `CRITICAL`)
- [ ] `tags` array starts with `"code-smell"`
- [ ] `impacts[0].softwareQuality` is `"MAINTAINABILITY"` or `"RELIABILITY"`
- [ ] `description` explains the smell, its consequences, and the violated design principle
- [ ] `remediation.examples` show recognizable anti-patterns and refactored solutions
- [ ] If the rule has a numerical threshold, `params` is defined with a sensible default
- [ ] If using `LINEAR` debt, both `coefficient` and `offset` are provided
- [ ] `key` matches the filename (without `.json`), lowercase hyphen-separated
- [ ] No overlap with the 13 existing rules listed above
- [ ] JSON is valid and all required fields are present
