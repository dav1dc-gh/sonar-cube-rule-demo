# Code Smells Rules — Path-Specific Custom Instructions

## Purpose

This directory contains **13 SonarQube code smell rules**. Each rule detects a design or quality problem that makes code harder to understand, change, or test. These are not bugs or vulnerabilities — they are indicators of structural weakness that increase long-term technical debt.

---

## Category Conventions

| Property | Expected Value |
|---|---|
| `type` | `CODE_SMELL` (always) |
| `severity` | `MAJOR` for design issues with high impact (god class, duplication, refused bequest, empty catch); `MINOR` for lower-impact smells |
| `impacts[].softwareQuality` | `MAINTAINABILITY` (primary); `RELIABILITY` for error-handling smells like empty-catch-block |
| `tags` | Must include `"code-smell"`. Add design-principle tags where relevant: `"solid"`, `"dry-principle"`, `"law-of-demeter"`, `"yagni"` |

## Guidance for Creating New Code Smell Rules

1. **Always use `"type": "CODE_SMELL"`.** If a rule detects a crash or data corruption risk, it belongs in a different category (`BUG`).
2. **Severity calibration:**
   - `MAJOR` — The smell actively harms readability, testability, or changeability. Examples: god class, duplication, empty exception handling, violations of SOLID principles.
   - `MINOR` — The smell is a style or design preference that matters at scale but doesn't block understanding. Examples: magic numbers, feature envy, long parameter lists.
3. **Use `LINEAR` debt when the effort scales with the problem size.** God class and complex methods use `LINEAR` because a 1000-line class takes more effort to fix than a 501-line class. Most smells use `CONSTANT_ISSUE` because the fix is per-occurrence.
4. **Add `params` when there's a configurable threshold.** Complexity limits, line counts, parameter counts, and chain lengths should always be parameterized. Use sensible defaults aligned with industry norms.
5. **Tag with the design principle violated.** Use `"solid"` (SRP, LSP), `"dry-principle"`, `"law-of-demeter"`, `"yagni"`, `"cohesion"`, `"coupling"` as appropriate.
6. **Remediation examples should show refactoring patterns.** Code smells are fixed by restructuring. Show the before (smelly code) and after (refactored code) with realistic class/method names.
7. **Remediation cost calibration for this category:**
   - 5min: Trivial rename or extract-constant (magic numbers, dead code removal)
   - 10–20min: Extract method, add logging, break a chain
   - 30–45min: Extract class, introduce value object, restructure inheritance
   - 1h+: Major decomposition (god class → multiple services)

When creating or updating rules, note which design principles apply:

| Principle | Relevant Rules |
|---|---|
| **Single Responsibility (SRP)** | god-class, shotgun-surgery |
| **Liskov Substitution (LSP)** | refused-bequest |
| **Don't Repeat Yourself (DRY)** | duplicate-code, data-clumps |
| **Law of Demeter** | message-chains, feature-envy |
| **YAGNI** | speculative-generality, dead-code |
| **Encapsulation** | primitive-obsession, long-parameter-list |

## Common Pitfalls

- **Don't overlap with maintainability rules.** `long-parameter-list` (code-smells) and `too-many-parameters` (maintainability) are related but have different defaults (5 vs 7) and framing. Check the maintainability directory before creating a rule that may already exist there under a different name.
- **Don't use `VULNERABILITY` or `BUG` type.** Even if empty catch blocks can lead to bugs, the *smell* itself is a `CODE_SMELL`. The downstream consequence (missed error) would be a separate rule.
- **Keep before/after examples language-diverse when possible.** The existing rules use Java, JavaScript, and pseudocode. Match the language to what best illustrates the smell.
