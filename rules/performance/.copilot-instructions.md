# Performance Rules — Path-Specific Custom Instructions

## Purpose

This directory contains **12 SonarQube performance rules**. Each rule detects a runtime inefficiency — patterns that waste CPU, memory, I/O, or database resources. Performance rules are unique in this repository because they span multiple SonarQube types: some are `CODE_SMELL` (inefficient patterns), while others are `BUG` (resource leaks that cause failures).

---

## Category Conventions

| Property | Expected Value |
|---|---|
| `type` | `BUG` for resource leaks and failures (connection-pool-exhaustion, memory-leaks, unbounded-collection-growth, unoptimized-regex); `CODE_SMELL` for inefficient patterns that degrade performance but don't crash |
| `severity` | `CRITICAL` for rules that cause outages or data loss (pool exhaustion, memory leaks, unbounded growth, N+1, unoptimized regex); `MAJOR` for significant inefficiencies; `MINOR` for optimizations |
| `impacts[].softwareQuality` | `RELIABILITY` for resource-leak bugs; `MAINTAINABILITY` for inefficient patterns |
| `tags` | Must include `"performance"`. Add domain tags: `"memory"`, `"database"`, `"optimization"`, `"gc"`, `"async"`, `"regex"` |

## Guidance for Creating New Performance Rules

1. **Choose `BUG` vs `CODE_SMELL` carefully.** Use `BUG` when the issue will eventually cause a failure (OOM, pool exhaustion, thread starvation, ReDoS). Use `CODE_SMELL` when performance is degraded but the application remains functional.
2. **Severity calibration:**
   - `CRITICAL` — Causes production outages under load (pool exhaustion, memory leaks, N+1 at scale, ReDoS).
   - `MAJOR` — Measurable performance degradation (object creation in hot paths, blocking I/O in async, string concat in loops).
   - `MINOR` — Micro-optimizations with marginal impact (boxing, collection initial capacity).
3. **Impacts mapping:**
   - `BUG` rules → `"softwareQuality": "RELIABILITY"`, `"severity": "HIGH"`
   - `CODE_SMELL` rules with `CRITICAL` severity → `"softwareQuality": "MAINTAINABILITY"`, `"severity": "HIGH"`
   - `CODE_SMELL` rules with `MAJOR` severity → `"softwareQuality": "MAINTAINABILITY"`, `"severity": "MEDIUM"`
   - `CODE_SMELL` rules with `MINOR` severity → `"softwareQuality": "MAINTAINABILITY"`, `"severity": "LOW"`
4. **Remediation examples must show measurable improvement.** Don't just say "use a better approach." Show the specific API, pattern, or data structure that fixes the problem (e.g., `StringBuilder`, `try-with-resources`, `LinkedHashMap` with eviction).
5. **Tag with the resource domain.** Performance rules affect specific resources. Use tags like `"memory"`, `"database"`, `"network"`, `"gc"`, `"cache"`, `"concurrency"`, `"regex"`.
6. **Debt function** should be `CONSTANT_ISSUE` for all current performance rules. `LINEAR` would apply only if fix effort scales with code size (none currently do).
7. **Remediation cost calibration for this category:**
   - 5–10min: Drop-in replacement (boxing → primitives, StringBuilder, pre-sized collection)
   - 20–30min: Resource management pattern (try-with-resources, lazy init, async rewrite)
   - 45min+: Architectural changes (bounded caches, query refactoring with JOINs)
8. **Consider cross-category overlap.** Performance issues sometimes overlap with security (ReDoS) or maintainability (long methods with nested loops). Place the rule where its *primary* impact is felt. Add a secondary tag for the other concern.

## Type Decision Tree

Use this to determine `type` for new performance rules:

```
Does the issue cause the application to crash, hang, or lose data under load?
├── YES → type: BUG, severity: CRITICAL
│         Examples: memory leak, pool exhaustion, unbounded growth, ReDoS
└── NO  → Does the issue cause measurable latency or throughput degradation?
          ├── YES → type: CODE_SMELL, severity: MAJOR or CRITICAL
          │         Examples: N+1 queries, blocking I/O in async, string concat
          └── NO  → type: CODE_SMELL, severity: MINOR
                    Examples: unnecessary boxing, sub-optimal collection choice
```

## Common Pitfalls

- **Don't classify all performance issues as `CODE_SMELL`.** Resource leaks and unbounded growth are `BUG` because they guarantee eventual failure, not just slowness.
- **Don't use `VULNERABILITY` type.** Even performance-related security issues like ReDoS should be typed as `BUG` in this directory. If the primary concern is security (an attacker intentionally triggering ReDoS), create a separate rule in `rules/security/`.
- **Don't forget concurrency context.** Some rules (synchronous-io-in-async, missing-lazy-initialization) have thread-safety implications. Mention this in the description when relevant.
- **Keep remediation examples JVM-focused but note alternatives.** Most existing examples use Java. When the concept is universal (e.g., string concat in loops), consider showing a second language.
