# Performance Rules — Copilot Custom Instructions

You are assisting with **performance-focused SonarQube rule definitions** in this directory. Every rule here identifies runtime efficiency problems — excessive resource consumption, unnecessary allocations, blocking operations, or algorithmic anti-patterns that degrade application throughput, latency, or scalability. Follow these instructions precisely when creating, editing, or explaining performance rules.

---

## Category Constraints

- **`type` is `"CODE_SMELL"` or `"BUG"`** — use `CODE_SMELL` for inefficiency patterns that waste resources but produce correct results (e.g., string concatenation in loop). Use `BUG` for patterns that cause observable failures at scale (e.g., memory leaks, connection pool exhaustion).
- **`severity` is typically `"MAJOR"` or `"CRITICAL"`** — use `CRITICAL` for issues that cause outages, OOM errors, or severe degradation under load (e.g., memory leaks, N+1 queries, connection pool exhaustion). Use `MAJOR` for inefficiencies that impact performance measurably but don't cause failures (e.g., unnecessary boxing, string concatenation).
- **`impacts[].softwareQuality`** should be `"RELIABILITY"` for `BUG`-type rules (memory leaks, resource exhaustion) or `"MAINTAINABILITY"` for `CODE_SMELL`-type rules. Impact severity should be `"HIGH"` for CRITICAL rules and `"MEDIUM"` for MAJOR rules.
- **`tags` MUST include `"performance"`** as the first tag. Add specific domain tags: `"memory"`, `"database"`, `"optimization"`, `"resource-management"`, `"strings"`, `"collections"`, `"concurrency"`, `"regex"`, `"orm"`.

## Performance-Specific Guidance

### Performance Domains

Performance rules span several distinct domains. When creating a new rule, identify which domain it belongs to and use appropriate tags:

- **Memory management** — object allocation, garbage collection pressure, memory leaks, resource cleanup (`"memory"`, `"resource-management"`)
- **Database / I/O** — query patterns, connection handling, blocking operations, N+1 problems (`"database"`, `"orm"`, `"io"`)
- **Data structures** — collection sizing, algorithmic complexity, inappropriate data structure choices (`"collections"`, `"optimization"`)
- **String handling** — concatenation, formatting, encoding inefficiencies (`"strings"`, `"memory"`)
- **Concurrency** — thread contention, blocking in async contexts, synchronization overhead (`"concurrency"`, `"async"`)
- **Compilation / Interpretation overhead** — repeated regex compilation, reflection abuse (`"regex"`, `"optimization"`)

### Choosing Between `BUG` and `CODE_SMELL`

This is the only category that uses both types. Apply this decision framework:

- **Use `BUG`** when the pattern causes **observable failures** under realistic conditions: `OutOfMemoryError`, connection timeouts, thread starvation, application crashes. Examples: memory leaks, connection pool exhaustion.
- **Use `CODE_SMELL`** when the pattern is **inefficient but functionally correct**: the code produces the right result, just slower or with more resource usage than necessary. Examples: string concatenation in loop, unnecessary boxing, N+1 queries.

### Writing Descriptions

Performance rule descriptions must explain:
1. **The anti-pattern** — what specific coding practice causes the performance problem.
2. **The performance impact** — how it manifests (CPU spikes, memory pressure, increased latency, I/O saturation).
3. **The scaling behavior** — whether the problem worsens linearly, quadratically, or exponentially with input size or load.

Example: _"Detects potential N+1 query issues where database queries are executed inside loops, causing severe performance degradation."_

### Writing Remediation Examples

Performance remediation examples must:
- Show a **measurably inefficient** pattern in `before` — ideally one where the performance difference is obvious from code inspection (e.g., query in a loop, concatenation in a loop).
- Show the **optimized alternative** in `after` — using the idiomatic, efficient approach (e.g., batch query with JOIN, `StringBuilder`, try-with-resources).
- Include enough context to show the **loop or scaling factor** that makes the anti-pattern harmful. A single-invocation snippet that "looks fine" is not useful.
- Use Java conventions by default.

### Remediation Cost

Performance fixes range from simple API replacements to architectural changes:
- Simple swap (StringBuilder, initial capacity, try-with-resources): `"10min"` – `"15min"`
- Query restructuring (JOIN, batch fetch, lazy init): `"20min"` – `"30min"`
- Architectural change (caching layer, async pipeline, connection pool tuning): `"30min"` – `"60min"`

Use `CONSTANT_ISSUE` debt function for most performance rules. Only use `LINEAR` if the fix cost scales with a measurable dimension of the issue.

### Parameters

Performance rules occasionally have configurable thresholds — for example, maximum collection size before requiring eviction, or a regex complexity score. Add `params` only when:
- There is a genuine numerical boundary that separates "acceptable" from "problematic."
- Different deployment contexts (high-throughput server vs. batch job) would justify different thresholds.

## Template for New Performance Rules

```json
{
  "key": "rule-key-here",
  "name": "Human-Readable Performance Issue Name",
  "description": "Detects [anti-pattern] that causes [performance impact] by [mechanism], particularly problematic when [scaling condition].",
  "severity": "MAJOR",
  "type": "CODE_SMELL",
  "tags": ["performance", "relevant-domain-tag", "optimization"],
  "remediation": {
    "constantCost": "15min",
    "examples": [
      {
        "before": "// Inefficient code showing the anti-pattern in a realistic context",
        "after": "// Optimized code using the idiomatic efficient approach"
      }
    ]
  },
  "impacts": [
    {
      "softwareQuality": "MAINTAINABILITY",
      "severity": "MEDIUM"
    }
  ],
  "defaultSeverity": "MAJOR",
  "status": "READY",
  "debt": {
    "function": "CONSTANT_ISSUE",
    "offset": "15min"
  }
}
```

For `BUG`-type performance rules (resource leaks, exhaustion), adjust the template:

```json
{
  "severity": "CRITICAL",
  "type": "BUG",
  "impacts": [
    {
      "softwareQuality": "RELIABILITY",
      "severity": "HIGH"
    }
  ],
  "defaultSeverity": "CRITICAL"
}
```

## Validation Checklist

Before saving a new or modified performance rule, verify:
- [ ] `type` is `"CODE_SMELL"` (inefficiency) or `"BUG"` (causes failures) — justified by the decision framework above
- [ ] `severity` is `"MAJOR"` or `"CRITICAL"` — matches the type and impact
- [ ] `tags` array starts with `"performance"` and includes relevant domain tags
- [ ] `impacts[0].softwareQuality` is `"RELIABILITY"` for BUG, `"MAINTAINABILITY"` for CODE_SMELL
- [ ] `description` explains the anti-pattern, the performance impact, and the scaling behavior
- [ ] `remediation.examples` show inefficient code with enough context to see the scaling problem, paired with the optimized alternative
- [ ] `key` matches the filename (without `.json`), lowercase hyphen-separated
- [ ] No overlap with the 12 existing rules listed above
- [ ] JSON is valid and all required fields are present
